================================================================================
О ПРОЕКТЕ
================================================================================

Название: Grind & Go HSE — мобильное приложение для заказа кофе (курсовой проект HSE).

Структура репозитория:
- grind_go_mobile/     — Flutter-приложение (клиент + интерфейс барista)
- GrindGoHSE/          — ASP.NET Core 9 Web API + PostgreSQL
- sql/create_db.sql    — SQL-схема базы данных

Стек:
- Flutter (Dart), provider, http, mask_text_input_formatter
- ASP.NET Core 9, EF Core, Npgsql, JWT, BCrypt, Swagger
- PostgreSQL (база coffee_shop_db)

API по умолчанию: http://localhost:5000
Swagger: http://localhost:5000/swagger (только Development)

--- 4. Отправка заказа барista ---

Запрос: добавить возможность отправки заказа барista.

Сделано (BACKEND):
- OrderStatuses, OrderService, OrdersController
- POST /api/orders (роль client) — создание заказа
- GET /api/orders/me — история заказов клиента
- GET /api/orders/queue (роль barista) — очередь заказов
- PATCH /api/orders/{id}/status — смена статуса (создан → в работе → готов к выдаче)
- SeedBaristaAsync: тестовый барista +79991112233 / barista123
- NotifyBaristaNewOrderAsync (заглушка NoOpNotificationService)

Сделано (MOBILE):
- Подключение к реальному API (http, ApiClient, ApiConfig)
- AuthRepository вместо MockAuthRepository
- MenuRepository — меню с /api/menu (productSizeId)
- OrderRepository — POST /api/orders
- CartScreen: кнопка «Заказать» отправляет заказ и очищает корзину
- HomeScreen загружает меню из API

--- 5. Проблемы с входом / регистрацией ---

Симптомы:
- ClientException: Failed to fetch (API не запущен или PostgreSQL недоступен)
- 28P01: password authentication failed for user postgres (неверный пароль в appsettings)
- «Неверный телефон или пароль» у барista

Причина формата телефона:
- В БД барista: +79991112233
- Приложение отправляло: +7 (999) 111-22-33

Исправлено:
- normalizePhone() в grind_go_mobile/lib/core/phone_input.dart
- AuthRepository использует normalizePhone при login/register
- AuthService.NormalizePhone на backend (strip нецифровых, формат +7XXXXXXXXXX)
- LoginScreen/RegistrationScreen: catch ApiException (не только MockAuthException)
- Пароль регистрации: минимум 6 символов (как на backend)

--- 7. Избранное (сердечки) — ДОБАВЛЕНО И ОТКАТАНО ---

Запрос: сохранять избранные напитки per user, показывать их первыми в меню.

Было реализовано:
- Таблица product_favorites, FavoritesController, FavoritesService
- GET/POST/DELETE /api/favorites
- FavoritesProvider, FavoritesRepository во Flutter
- Сортировка меню, сердечко на карточках и детальном экране

Проблема: ручная миграция без обновления AppDbContextModelSnapshot →
  PendingModelChangesWarning при dotnet run

Запрос пользователя: «Верни всё в изначальный вид до запроса про сердечки»

Откат выполнен:
- Удалены все файлы избранного (backend + mobile)
- Сердечко снова локальное на ProductDetailScreen (_isFavorite, не сохраняется)
- Меню без сортировки по избранному
- Миграция AddProductFavorites удалена из кода

Примечание: если таблица product_favorites уже создана в PostgreSQL, она может
остаться в БД неиспользуемой — на работу не влияет.

ИЗВЕСТНЫЕ ПРОБЛЕМЫ / ЗАМЕТКИ
================================================================================

1. PostgreSQL: пароль в appsettings.json должен совпадать с локальным postgres.
2. API должен быть запущен до работы Flutter-приложения.
3. dotnet-ef может быть не установлен глобально; миграции создавались вручную
   для orders — для favorites миграция была удалена при откате.
4. Фото напитков и админка для редактирования меню — не реализованы.
5. Push-уведомления — заглушка NoOpNotificationService.
6. Корзina и заказы привязаны к productSizeId из API (не mock id).
